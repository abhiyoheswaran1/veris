# VerisKit v0.5 — Design Spec (CI/PR Publishing + Browser Execution + History)

**Status:** Approved (design)
**Date:** 2026-07-13
**Branch:** `feat/veris-v0.5`
**Builds on:** v0.4.1 (shipped, `veriskit@0.4.1`). Reuses the verdict/record/reporter/runner/orchestrator/git seams.

---

## 1. Thesis

Three independent subsystems ship together as v0.5.0, each cleanly bounded and independently testable:

- **A. CI / PR publishing** — surface the honest verdict where teams review: a GitHub PR comment, a Check Run, and a status badge. All opt-in.
- **B. Browser execution** — turn the detected-but-deferred `browser` capability into a real Playwright runner folded into the verdict.
- **C. History** — `veris log` turns the evidence records already written into a trend, and flags flaky checks.

Still exactly two runtime dependencies (`cac` + `picocolors`). The GitHub API goes over Node's built-in `fetch`. Still honest: publishing is a side channel that never changes the verdict; browser flakiness is surfaced; flaky detection is defined plainly; history is local per-machine and said so.

## 2. Global constraints

| Constraint | Value |
|---|---|
| Runtime deps | exactly `cac` + `picocolors`; GitHub API over built-in `fetch`; no new dep |
| Command / brand | command `veris`; brand string `VerisKit` |
| Module style | ESM, `.js` import specifiers |
| Secrets | `GITHUB_TOKEN` read from env only; never stored, printed, logged, or embedded |
| Honesty | `--github` never fails the build on a publish error; the verdict/exit code come only from the checks |
| Docs prose | no em-dashes (code fences exempt) |
| Deliverable | `veriskit@0.5.0`, single publish via the existing OIDC pipeline |

## 3. Subsystem A: CI / PR publishing (`src/publish/`)

### Scope
- `veris verify --github` and `veris affected --github`: when a GitHub PR context is present, publish the verdict to a sticky PR comment and a Check Run.
- `veris badge [--out <path>]`: write a shields.io endpoint JSON from the latest verdict.

### Modules
```
src/publish/
  context.ts   # PublishContext | null from GITHUB_* env
  github.ts    # fetch client: upsertComment, createCheckRun
  comment.ts   # render the sticky comment body
  badge.ts     # shields endpoint JSON from a verdict
```

### 3.1 context.ts
`resolvePublishContext(env = process.env): PublishContext | null` returns `null` unless all of the following resolve:
- `repo`: from `GITHUB_REPOSITORY` (`"owner/name"`), split into `{ owner, repo }`.
- `token`: from `GITHUB_TOKEN`.
- `prNumber`: parse `GITHUB_REF` of the form `refs/pull/<n>/merge` (or `/head`); if absent, read `GITHUB_EVENT_PATH` JSON and take `pull_request.number`. `null` when neither yields a PR.
- `headSha`: `pull_request.head.sha` from the event payload when available, else `GITHUB_SHA`.

```ts
export interface PublishContext {
  owner: string;
  repo: string;
  token: string;
  prNumber: number;
  headSha: string;
}
```

A missing token, repo, or PR yields `null` (not an error): the caller prints a notice and skips publishing.

### 3.2 github.ts
A thin `fetch` client. All requests send `Authorization: Bearer <token>`, `Accept: application/vnd.github+json`, `X-GitHub-Api-Version: 2022-11-28`, `User-Agent: veriskit`.
- `upsertComment(ctx, body): Promise<void>` — `GET /repos/{owner}/{repo}/issues/{prNumber}/comments` (paginated first page is enough for the common case), find the first comment whose body contains the marker `<!-- veriskit-report -->`; if found `PATCH /repos/{owner}/{repo}/issues/comments/{id}`, else `POST /repos/{owner}/{repo}/issues/{prNumber}/comments`.
- `createCheckRun(ctx, { name, conclusion, title, summary }): Promise<void>` — `POST /repos/{owner}/{repo}/check-runs` with `head_sha`, `status: "completed"`, `conclusion`, and `output: { title, summary }`.
- Non-2xx responses throw a `GitHubApiError` carrying the status and a short message. The token is never included in any thrown message or log.

### 3.3 comment.ts
`renderComment(run: VerificationRun): string` returns the marker line `<!-- veriskit-report -->` followed by a one-line verdict header (state + per-check summary) and the full Markdown report inside a `<details>` block. Reuses `renderMarkdown` for the report body.

### 3.4 badge.ts
`buildBadge(verdict: Verdict): { schemaVersion: 1; label: string; message: string; color: string }` maps `verified → "verified"/"14b8a6"`, `failed → "failed"/"e5484d"`, `partial → "partial"/"f5a623"`. `veris badge` reads the latest evidence record (via `latestRunDir`), builds the badge, and writes it to `--out` (default `.veris/badge.json`).

### 3.5 Check Run conclusion mapping
`verified → success`, `failed → failure`, `partial → neutral`.

### 3.6 CLI wiring
- `verify` and `affected` gain a `--github` boolean option. After the verdict + record are built and the terminal output is printed, if `--github` is set: `resolvePublishContext()`; if `null`, print `veris: --github set but no GitHub PR context found (need GITHUB_TOKEN + a pull_request event); skipping publish.` and continue; else call `upsertComment` and `createCheckRun`, each wrapped so a thrown error prints `veris: GitHub publish failed: <message>` and does NOT change `process.exitCode`.
- New `badge` command: `veris badge [--out <path>]`.
- **Token permissions:** posting the comment needs `pull-requests: write` and creating the Check Run needs `checks: write`. The README documents adding a `permissions:` block to the workflow. If the API returns 403 for a missing permission, VerisKit reports it (per the honesty rule) and the verdict is unchanged.

### 3.7 Testing (A)
- `resolvePublishContext`: full env → context; missing token/repo/PR → `null`; `refs/pull/42/merge` → `prNumber 42`.
- `github.ts`: with a mocked global `fetch`, `upsertComment` PATCHes when a marked comment exists and POSTs when not; `createCheckRun` posts the right conclusion; a 403 response throws `GitHubApiError` whose message excludes the token.
- `badge.ts`: each verdict state maps to the right label/color; the JSON has `schemaVersion: 1`.
- `comment.ts`: body starts with the marker and contains the verdict and a `<details>` block.
- CLI: `verify --github` with no context prints the notice and leaves the exit code equal to the verdict's.

## 4. Subsystem B: Browser execution (`src/runners/playwright.ts`)

### Scope
Turn `browser` from "detected, deferred" into a real capability run by a Playwright runner, opt-in (not in the default check set).

### 4.1 The runner
`playwrightRunner` follows the existing `vitestRunner` shape (`src/runners/base.ts` + `src/runners/vitest.ts`):
- `toCheck(project, cap, opts?)`: `cmd = localBin(project.root, "playwright")`, `args = ["test", "--reporter=json"]`.
- `run(check, ctx)`: exec the command; parse the Playwright JSON reporter from stdout. Playwright's JSON reporter emits a top-level `stats: { expected, unexpected, flaky, skipped }`. Map: `passed = exit code 0 && stats.unexpected === 0`; `counts = { passed: stats.expected, failed: stats.unexpected, total: stats.expected + stats.unexpected + stats.flaky + stats.skipped }`. On a non-zero exit with unparseable output, status `failed` with the output tail. Write the raw output to the log via `writeLog`.
- Register in `src/runners/index.ts` as `playwright`.

### 4.2 Detection
`detectBrowser` in `src/config/detect.ts`: when `@playwright/test` is a dependency or `playwright.config.ts`/`playwright.config.js` exists, return `{ id: "browser", available: true, runner: "playwright" }` (remove the "deferred to v0.5" reason). Otherwise unchanged (`available: false`).

### 4.3 Opt-in
`browser` stays out of `DEFAULT_CHECKS` (`["types", "lint", "unit"]`). `verify` gains a `--browser` flag that appends `"browser"` to the requested checks when the capability is available; a project may also list `browser` in `.veris/config.json` `checks`. When not requested, browser is simply not run and not counted (honest: an un-requested check does not lower the verdict). `doctor` shows `browser` as available with runner `playwright`.

### 4.4 Testing (B)
- `playwrightRunner.toCheck` builds `playwright test --reporter=json` against the local bin.
- `run` parses a sample Playwright JSON stats blob into the right `status`/`counts`; a non-zero exit with unexpected>0 → `failed`.
- `detectBrowser` returns `available: true, runner: "playwright"` when a `playwright.config.ts` is present (temp dir), `available: false` otherwise.
- `verify --browser` adds `browser` to the requested set only when available (unit-level check of the check-selection logic).

## 5. Subsystem C: History (`src/history/`)

### Scope
- `veris log [--limit N]` — list past runs newest-first from stored evidence records.
- `veris log --flaky` — flag checks that flip-flop across the recent window.

### Modules
```
src/history/
  read.ts    # loadRuns(root, limit?) -> EvidenceRecord[] (newest first)
  flaky.ts   # detectFlaky(records) -> FlakyCheck[]
src/cli/commands/log.ts
```

### 5.1 read.ts
`loadRuns(root: string, limit = 20): EvidenceRecord[]` reads every `.veris/runs/*/evidence.json` (skips `watch` and unreadable/invalid files), sorts by `startedAt` descending, and returns the first `limit`. Uses the `EvidenceRecord` type and schema `veriskit/evidence@1` from `src/evidence/record.ts`.

### 5.2 flaky.ts
`detectFlaky(records: EvidenceRecord[]): FlakyCheck[]` where `FlakyCheck = { id: CapabilityId; statuses: CheckStatus[] }`. For each capability id appearing in the records' `checks`, collect its `status` across the window (newest first). A capability is flaky when its collected statuses include BOTH `passed` and `failed` (it changed outcome across runs without a code reason we can see). `skipped`/`unknown` are ignored for the flip-flop test. Returns only flaky capabilities.

### 5.3 cli/commands/log.ts
`runLog(root, opts: { limit?: number; flaky?: boolean }): Promise<number>`:
- default: print each run as `<date>  <verdict>  <check:status ...>  <commit7>`, newest first; if none, print `No runs yet. Run \`veris verify\` first.`
- `--flaky`: print the flaky capabilities and their status sequence, or `No flaky checks in the last N runs.`; print the one-line definition of "flaky" used.
- Read-only. States that history is local (the `.veris/runs` dir is gitignored, so this is per-machine).

### 5.4 Testing (C)
- `loadRuns` sorts newest-first, respects `--limit`, and skips an invalid/`watch` entry.
- `detectFlaky`: a capability that is passed, then failed, then passed → flagged; an always-passed capability → not flagged; a skipped-only capability → not flagged.
- `runLog` renders a list and the flaky report (exit 0 in both, and 0 when there are no runs).

## 6. Cross-cutting

- README gains three sections (Publish, Browser tests, History); CHANGELOG `## 0.5.0`; version bumped to `0.5.0`.
- The stale `detectBrowser` "deferred to v0.5" note is removed (subsystem B ships it).
- Dogfooding: A is exercised with mocked `fetch` in tests; the live PR-posting path is verified by the user in a real CI run, not fabricated. B is dogfooded only if Playwright is present (VerisKit's own repo has none, so B's dogfood is the unit tests plus `doctor` showing the capability shape). C is dogfooded against VerisKit's own accumulated `.veris/runs`.

## 7. Out of scope (deferred)

- GitLab / Bitbucket publishing; only GitHub in v0.5.
- Posting to non-PR contexts (pushes to a branch): `--github` only publishes in a PR context.
- Trend graphs, HTML history, or persisting history outside `.veris/runs`.
- Running browser tests by default; sharding or per-project Playwright configuration beyond invoking `playwright test`.
- Sigstore keyless signing (still deferred from v0.4.1).

## 8. Deliverable

`veriskit@0.5.0`: `--github` publishing (PR comment + Check Run) and `veris badge`; a real Playwright browser runner (opt-in via `--browser`); `veris log` with `--flaky`; README + CHANGELOG updated; dogfooded per section 6; published via the OIDC release pipeline.
