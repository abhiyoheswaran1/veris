# VerisKit v0.4 — Design Spec ("Evidence System")

**Status:** Approved (design)
**Date:** 2026-07-10
**Branch:** `feat/veris-v0.4`
**Builds on:** v0.1 + v0.2 + v0.3 (shipped, `veriskit@0.3.0`). Reuses the detect / orchestrator / verdict / reporter / evidence / runner / git seams. Upgrades v0.1's local-only evidence (`metadata.json` + Markdown report + raw logs) into a canonical, tamper-evident, portable evidence record.

---

## 1. Thesis

VerisKit's brand is *"can we trust the result?"* Today a run leaves a Markdown report and a `metadata.json`, both plain files anyone can silently edit, neither tied to a specific commit. v0.4 makes every run leave a **canonical, versioned, git-anchored evidence record** carrying a reproducible sha256 integrity digest, plus a **single-file portable bundle** you can hand to someone as proof a run passed.

Still exactly two runtime dependencies (`cac` + `picocolors`); the digest comes from node's built-in `crypto`, the git anchor from the existing `git/changes.ts` seam. Still honest: the digest is labeled an **integrity digest**, never "tamper-proof"; a dirty working tree is recorded and surfaced, never hidden.

One new command group (`veris evidence` with `verify` / `bundle` / `show`) plus an upgrade to what `verify` / `affected` / `watch` write on every run.

## 2. Locked decisions

| Decision | Choice | Rationale |
|---|---|---|
| Core job | **Trustworthy + portable record** | Directly serves the product thesis. History/trends and CI/PR publishing deferred to later v0.4.x |
| Integrity mechanism | **sha256 integrity digest + git anchor** (commit SHA, branch, dirty flag) | No new deps, works offline, ties the verdict to exact code. Honestly labeled: detects corruption and edits, not forgery-proof on its own |
| Cryptographic signing | **Deferred to v0.4.x** (sigstore keyless) | Would add a heavy dependency + network/OIDC requirement, breaking the two-dep budget |
| Bundle format | **Single self-contained JSON file**, not `.tgz` | A tarball needs a third runtime dependency. One JSON file with embedded logs is portable and dep-free |
| `metadata.json` | **Superseded by `evidence.json`** (strict superset) | Avoid two overlapping files; the Markdown report is generated from the evidence record |

## 3. Scope

### In
- Canonical evidence record `src/evidence/record.ts`: build from a `VerificationRun` + git anchor, deterministic canonicalization, sha256 digest (`node:crypto`).
- `evidence.json` written on every `verify` / `affected` / `watch` run to `.veris/runs/<id>/evidence.json` (replaces `metadata.json`).
- Per-check `logDigest` (sha256 of each raw `.log`).
- `git/changes.ts` gains a `gitAnchor()` helper: commit SHA, branch, dirty flag, changed-file count.
- `veris evidence verify <path>` — recompute + compare digest and per-log digests; exit 0 on match, 1 on mismatch; honest labeling of what it proves.
- `veris evidence bundle [--out <file>]` — single-file portable proof (record + embedded logs + report + per-file manifest + top-level bundle digest).
- `veris evidence show [<path>]` — pretty-print the latest record's key facts.
- Markdown report generated from the evidence record; report gains the git anchor + digest lines.
- README/CHANGELOG updates; dogfood; publish `veriskit@0.4.0`.

### Out (deferred — do NOT build in v0.4)
- Cryptographic signing (sigstore keyless) → **v0.4.x**.
- Evidence history / ledger / `veris log` / flakiness detection → later v0.4.x.
- CI / PR publishing (GitHub check run, PR comment, status badge) → later v0.4.x.
- Any network calls, remote sync, or hosted evidence store.

## 4. Architecture

New modules; existing modules consumed unchanged except `store.ts` (writes `evidence.json` instead of `metadata.json`), `git/changes.ts` (gains `gitAnchor()`), and the reporters (read the record, add git + digest lines).

```
src/
  evidence/
    record.ts           # EvidenceRecord type; buildRecord(run, anchor); canonicalize(); computeDigest()
    bundle.ts           # buildBundle(record, logs, report) -> single JSON; readBundle()
    verify-evidence.ts  # recompute digest + per-log digests, compare, produce a VerifyResult
    store.ts            # (extended) writeEvidence() replaces writeMetadata()
  git/
    changes.ts          # (extended) gitAnchor(): { commit, branch, dirty, changedFiles }
  cli/commands/
    evidence.ts         # `veris evidence verify|bundle|show` subcommands
  reporters/
    markdown.ts         # (extended) render git anchor + digest section
    terminal.ts         # (extended) print commit + tree state line
```

### 4.1 The evidence record (`veriskit/evidence@1`)

```jsonc
{
  "schema": "veriskit/evidence@1",
  "id": "2026-07-10T14-10-01-796Z-1",
  "startedAt": "2026-07-10T14:10:01.796Z",
  "tool": { "name": "veriskit", "version": "0.4.0" },
  "git": { "commit": "4fa33a9...", "branch": "main", "dirty": false, "changedFiles": 0 },
  "env": { "os": "...", "node": "...", "pm": "...", "ci": false, "timestamp": "..." },
  "project": { "name": "veriskit", "packageManager": "npm", "frameworks": [], "languages": [] },
  "scope": { "kind": "full", "changedCount": 0 },
  "checks": [
    { "id": "unit", "status": "passed", "runner": "vitest", "durationMs": 13300,
      "summary": "81 passed", "counts": { "passed": 81, "total": 81 },
      "logDigest": "sha256:..." }
  ],
  "verdict": { "state": "verified", "verifiedCapabilities": ["types","unit","lint"],
               "skipped": [], "reasons": [] },
  "digest": "sha256:9f2c..."
}
```

- `scope.kind` is one of `full | affected | watch`.
- `checks[].logDigest` is present only when the check wrote a log.
- `digest` is computed last, over the canonical form of the record with the `digest` key omitted.

### 4.2 Canonicalization (`canonicalize`)

Deterministic serialization so the digest reproduces byte-for-byte:
- Recursively sort object keys ascending.
- Arrays keep their given order (checks stay in run order).
- `JSON.stringify` with no indentation (no insignificant whitespace).
- UTF-8 bytes are the digest input.

`computeDigest(record)` = `"sha256:" + sha256(canonicalize({ ...record, digest: undefined }))`, using `node:crypto`'s `createHash("sha256")`.

### 4.3 Git anchor (`gitAnchor`)

Reuses the existing `child_process` git calls in `git/changes.ts`:
- `commit`: `git rev-parse HEAD` (store the full SHA; the terminal and report shorten it for display).
- `branch`: `git rev-parse --abbrev-ref HEAD`.
- `dirty`: `git status --porcelain` non-empty.
- `changedFiles`: count of porcelain lines.
- Outside a git repo or with no commits: `git` is `null` in the record (honestly "no git anchor available"), and the record still validates.

### 4.4 Bundle (`veris evidence bundle`)

A single JSON document, schema `veriskit/bundle@1`:

```jsonc
{
  "schema": "veriskit/bundle@1",
  "record": { /* the full evidence record, verbatim */ },
  "report": "# VerisKit Verification Report\n...",     // the Markdown, inline
  "logs": { "unit": "<raw log text>", "types": "..." }, // per-check raw logs, inline
  "manifest": {
    "record": "sha256:...",   // digest of the canonical record (== record.digest)
    "report": "sha256:...",   // digest of the report string
    "logs": { "unit": "sha256:...", "types": "..." }
  },
  "bundleDigest": "sha256:..." // over the canonical bundle with bundleDigest omitted
}
```

Default output path `.veris/evidence/<id>.bundle.json`. `veris evidence verify` accepts either a raw `evidence.json` or a bundle: for a bundle it checks the record digest, every manifest entry against the inline content, and the top-level `bundleDigest`.

### 4.5 `veris evidence verify` output (honest labeling)

```
digest OK · commit 4fa33a9 · tree clean
verdict: verified (types, unit, lint)

This confirms the record has not been edited or corrupted since it was written.
A self-contained digest is not forgery-proof on its own: to prove authorship,
publish the digest separately (CI log, PR) or sign it (planned, v0.4.x).
```

Mismatch: name exactly what failed (record digest, a specific log digest, or bundle digest), exit 1.

## 5. Honesty guarantees (the brand)

- The digest is called an **integrity digest**, never "tamper-proof." `evidence verify` states what it does and does not prove.
- A dirty tree is recorded (`dirty: true`) and surfaced in the terminal, report, and `evidence verify`. The verdict then reads "these results, on commit X plus uncommitted edits."
- Canonicalization is deterministic; a round-trip (build → digest → reverify) must always pass. If a value cannot be canonicalized deterministically, fail loudly rather than emit a digest that will not reverify.
- No network calls, no telemetry, no hidden state. Everything stays under `.veris/`.

## 6. Testing

- **Round-trip:** build record → digest → `verify-evidence` → OK. Mutate any field → verify fails, naming the record digest.
- **Determinism:** the same `VerificationRun` + anchor produces byte-identical canonical output and an identical digest across two independent builds.
- **Key ordering:** records that differ only in input key insertion order produce the same digest.
- **Git anchor:** clean vs dirty tree recorded correctly; no-git case yields `git: null` and still validates (stub the git calls).
- **Per-log digest:** tamper an embedded log in a bundle → `verify` fails, naming that log.
- **Bundle:** assemble → verify manifest + bundleDigest → OK; tamper the report string → fails.
- **Backward-compat:** the Markdown report is still generated correctly, now from the evidence record, and includes the git + digest lines.
- **CLI:** `evidence verify` exit codes (0 match / 1 mismatch); `evidence bundle` writes the default path; `evidence show` prints key facts.

## 7. Deliverable

`veriskit@0.4.0`: every run writes a canonical, git-anchored `evidence.json`; `veris evidence verify|bundle|show`; report + terminal show the git anchor and digest; README + CHANGELOG updated; dogfooded on VerisKit's own repo.
