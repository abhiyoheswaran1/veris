# VerisKit v0.6 — Design Spec ("MCP server")

**Status:** Approved (design)
**Date:** 2026-07-14
**Branch:** `feat/veris-v0.6`
**Builds on:** v0.5.1 (shipped, `veriskit@0.5.1`). Reuses the whole core (detect / orchestrator / verdict / record / project-graph / history / evidence seams) through a new programmatic API.

---

## 1. Thesis

Let AI agents call VerisKit directly. A new `veriskit-mcp` package exposes VerisKit's verification as Model Context Protocol tools over stdio, publishable to the MCP registry, so an agent can ask "is this change safe to trust?" as a tool call.

The `veriskit` CLI keeps its brand promise of **exactly two runtime dependencies** (`cac` + `picocolors`). The MCP server is a separate package: it takes `@modelcontextprotocol/sdk` and depends on `veriskit` for the core. The SDK lives only in `veriskit-mcp`.

Still honest: every tool returns the real three-state verdict and the skip reasons; nothing is folded or hidden.

## 2. Locked decisions

| Decision | Choice | Rationale |
|---|---|---|
| Packaging | **Separate `veriskit-mcp` package** using the official SDK, depending on `veriskit` | Keeps the CLI's two-dep promise; the SDK tracks the evolving MCP protocol |
| Repo layout | **npm workspaces**: root `veriskit` + a new `mcp/` workspace | Adds the package without a disruptive `packages/` restructure of the already-published root |
| Core access | **`veriskit` gains a programmatic API export** (`src/index.ts`) | The MCP server imports typed core functions instead of shelling out or reaching into `dist` |
| Transport | **stdio only** | The registry's common case; HTTP/SSE deferred |
| Tool set | **7 tools** (doctor, scan, plan, log, evidence_verify, verify, affected) | The user's six plus `doctor` so an agent can ask what is runnable |
| Registry namespace | **`io.github.abhiyoheswaran1/veriskit`** | GitHub auth, no DNS. Switchable to `com.baseframelabs/veriskit` (DNS TXT) later |
| Project root | **optional `path` arg, default the server's launch cwd** | Works whether the client sets cwd or passes a path |

## 3. Scope

### In
- `veriskit` programmatic API: `src/index.ts` re-exporting the core the MCP server needs, plus `package.json` `main`/`module`/`types`/`exports` and a second tsup entry. No new `veriskit` dependency.
- Extract the verify and affected **orchestration** out of the CLI command functions into shared core functions `verifyProject(root, opts)` and `affectedProject(root, opts)` (each returns `{ run: VerificationRun, record: EvidenceRecord, note?: string }`). `runVerify`/`runAffected` become thin: call the core function, print, return the exit code. The MCP `veris_verify`/`veris_affected` tools call the same core functions, so a CLI verify and an MCP verify are identical. This is the one targeted refactor; no behavior change to the CLI.
- npm workspaces: root `package.json` gains `"workspaces": ["mcp"]`.
- `mcp/` package `veriskit-mcp`: `@modelcontextprotocol/sdk` (+ its `zod` schema peer) and `veriskit` as dependencies; a `veriskit-mcp` bin; an stdio `McpServer` registering the 7 tools; handlers that call `veriskit`'s core and return structured JSON.
- `mcp/server.json` registry manifest + `"mcpName"` in `veriskit-mcp`'s `package.json`.
- Release pipeline publishes both packages; README "Use with AI agents (MCP)" section + client config snippet; CHANGELOG; version 0.6.0 for both.

### Out (deferred)
- HTTP / SSE transport; MCP resources, prompts, sampling, or roots negotiation (tools only).
- Auto-signing, GitLab/Bitbucket, or any feature not already in `veriskit`.
- The actual `mcp-publisher` registry submission (a human step; documented, not automated here).
- Branded `com.baseframelabs/veriskit` namespace + DNS auth (a later switch).

## 4. Architecture

```
(root) veriskit                 CLI, deps unchanged (cac + picocolors)
  src/index.ts                  NEW: re-export the core API (types + functions)
  package.json                  NEW: "exports"/"main"/"module"/"types"; "workspaces": ["mcp"]
  tsup.config                   NEW second entry: src/index.ts -> dist/index.js
mcp/
  package.json                  veriskit-mcp; deps: @modelcontextprotocol/sdk, zod, veriskit
  bin/veriskit-mcp              #!/usr/bin/env node -> dist/server.js
  src/server.ts                 build McpServer, register tools, connect StdioServerTransport, main()
  src/tools.ts                  the 7 tool definitions (schema + annotations + handler)
  src/format.ts                 shape core results into tool JSON payloads
  src/tools.test.ts             handler unit tests
  src/server.test.ts           in-memory Client<->Server wiring test
  server.json                   registry manifest
  tsconfig.json / tsup config
```

### 4.1 The `veriskit` programmatic API (`src/index.ts`)

Re-exports the functions the MCP server calls and the types it returns:
`verifyProject`, `affectedProject`, `detectProject`, `buildGraph`, `analyze`, `loadRuns`, `detectFlaky`, `verifyEvidenceFile`, `getEnvironmentInfo`, plus `Project`, `VerificationRun`, `EvidenceRecord`, `Verdict`, `CapabilityId`, `ProjectGraph`, `Analysis`, `VerifyResult`, `FlakyCheck`. (`verifyProject`/`affectedProject` are the extracted orchestration; the lower-level `runChecks`/`buildRecord`/`gitAnchor` stay internal.) `package.json` gains `"exports": { ".": { "types": "./dist/index.d.ts", "import": "./dist/index.js" }, "./package.json": "./package.json" }` and keeps the existing `bin`.

### 4.2 Tools

Every handler resolves the project root as `input.path ?? process.cwd()`, calls the core, and returns `{ content: [{ type: "text", text: <pretty JSON> }] }`. The JSON is the structured result an agent can reason over.

| Tool | Input | Core calls | Result JSON |
|---|---|---|---|
| `veris_doctor` | `{ path? }` | `detectProject`, `getEnvironmentInfo` | capabilities (id, available, runner, reason), environment |
| `veris_scan` | `{ path? }` | `buildGraph`, `analyze` | resolver, counts, top untested high-impact files |
| `veris_plan` | `{ path?, base? }` | `buildGraph`, `analyze`, `changedFiles` | prioritized recommendations |
| `veris_log` | `{ path?, limit?, flaky? }` | `loadRuns`, `detectFlaky` | run list (date, verdict, checks, commit) or flaky checks |
| `veris_evidence_verify` | `{ file }` | `verifyEvidenceFile` | ok, per-check results, signed, verdict, git anchor |
| `veris_verify` | `{ path?, partialOk?, browser? }` | `detectProject`, `runChecks`, `gitAnchor`, `digestLogs`, `buildRecord`, write evidence | verdict, per-check results, digest, report markdown |
| `veris_affected` | `{ path?, base? }` | affected selection + `runChecks` + record | scoped verdict, narrowed note, per-check results |

Annotations: `veris_doctor|scan|plan|log|evidence_verify` set `readOnlyHint: true`. `veris_verify|affected` set `readOnlyHint: false` and their descriptions state they execute the project's test tooling and write evidence under `.veris/`. `veris_verify`/`affected` reuse the same check-selection and evidence-writing the CLI uses, so an MCP verify and a CLI verify agree.

### 4.3 Server (`server.ts`)

Creates an `McpServer` (name `veriskit`, version from `veriskit-mcp`'s package.json), registers the 7 tools via the SDK's `registerTool`, connects a `StdioServerTransport`, and starts. `main()` handles the connect and logs startup to stderr only (stdout is the protocol channel and must stay clean). A malformed tool input is rejected by the SDK's schema; a core error is caught and returned as an error tool result (`isError: true`) with a short message, never a crash.

### 4.4 Registry manifest (`server.json`)

```jsonc
{
  "$schema": "https://static.modelcontextprotocol.io/schemas/2025-07-09/server.schema.json",
  "name": "io.github.abhiyoheswaran1/veriskit",
  "description": "Verify software and check evidence with VerisKit, over MCP.",
  "version": "0.6.0",
  "packages": [
    { "registryType": "npm", "identifier": "veriskit-mcp", "version": "0.6.0",
      "transport": { "type": "stdio" } }
  ]
}
```

`veriskit-mcp`'s `package.json` carries `"mcpName": "io.github.abhiyoheswaran1/veriskit"` so the registry can verify the npm publisher owns the server name. The exact `server.json` schema evolves; validate against the current registry docs at publish time.

## 5. Honesty and safety

- Tools return the real verdict (`verified` / `failed` / `partial`) and the skipped list with reasons; a partial is never presented as a pass.
- `veris_verify` / `veris_affected` execute the project's configured test runners. Their descriptions say so, and their annotations mark them non-read-only, so an agent (and its human) know these run code, not just read files.
- The server writes nothing to stdout except MCP protocol frames; diagnostics go to stderr.
- `veriskit-mcp` adds the SDK; `veriskit` stays exactly `cac` + `picocolors`. Docs state this split plainly.

## 6. Testing

- **Tool handlers:** against a temp project seeded with a couple of source/test files: `veris_doctor` reports capabilities; `veris_scan` returns a graph summary; `veris_log` reads seeded evidence; `veris_evidence_verify` passes a good record and fails a tampered one. `veris_verify` runs on a tiny project and returns a `verified` verdict JSON.
- **Server wiring:** link an SDK `Client` to the `McpServer` over the in-memory transport pair; assert `tools/list` returns the 7 tools with correct annotations, and a `tools/call` of `veris_doctor` returns the expected JSON.
- **`veriskit` API export:** import from the built `veriskit` entry and run `runChecks`/`buildGraph` programmatically to confirm the public surface works.
- **No stdout noise:** a test that the server writes only protocol frames to stdout during a tool call (diagnostics on stderr).

## 7. Deliverable

`veriskit@0.6.0` (adds the library export, deps unchanged) and `veriskit-mcp@0.6.0` (the MCP server), both published via the release pipeline. README documents "Use with AI agents (MCP)" with a client config snippet. `mcp/server.json` + `mcpName` are in place so the user can run `mcp-publisher login` + `publish` to list it in the MCP registry. Dogfooded: the server starts, `tools/list` returns the 7 tools, and a `veris_verify` tool call on VerisKit's own repo returns a verified verdict.
