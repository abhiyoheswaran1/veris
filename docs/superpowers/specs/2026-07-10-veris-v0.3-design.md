# Veris v0.3 — Design Spec ("Project Intelligence")

**Status:** Approved (design)
**Date:** 2026-07-10
**Branch:** `feat/veris-v0.3`
**Builds on:** v0.1 + v0.2 (shipped, `veriskit@0.2.0`). Reuses the detect / orchestrator / verdict / reporter / evidence / runner seams. Upgrades v0.2's coarse `affected` to a precise, graph-based version.

---

## 1. Thesis

Give Veris a real map of the codebase — an import graph — so it can answer
*"what is actually tested, what is risky, and what should I test next?"* — and make
`affected` **precise** (changed file → the specific test files that reach it) instead of
coarse. Still zero runtime dependencies. Still honest: it always reports which resolver
built the graph, and it never claims coverage or narrows a test run it cannot prove.

Two new commands (`scan`, `plan`) plus a graph-based upgrade to `affected`/`watch`.

## 2. Locked decisions

| Decision | Choice | Rationale |
|---|---|---|
| Graph resolver | **Project's own `typescript`** (`ts.preProcessFile` + `ts.resolveModuleName`), with a **dep-free scanner** fallback | Accurate: honors tsconfig `paths`/`baseUrl`, barrel re-exports, type-only imports. Zero deps added (uses the project's TS, same "orchestrate what's there" ethos). Scanner covers plain-JS projects |
| Scope | **Graph core** (engine + `scan` + `plan` + graph-based `affected`). Route/endpoint mapping **deferred** to v0.3.x | Highest value, coherent; avoids fragile framework-specific heuristics |
| `plan` | **Analysis + recommendations only. No test generation** | Generation stays v0.8 |
| Graph-`affected` safety | **Conservative fallback** — narrow the unit run only when the graph is certain; else run the full suite | Speed must never cost honesty: an affected-but-unrun test can never be hidden |

## 3. Scope

### In
- Import-graph engine (`src/project-graph/`) with `ts-resolver` + `scanner-resolver`.
- `veris scan` — build + report the graph; write `.veris/graph.json` evidence.
- `veris plan` — prioritized recommendations from the graph (analysis only).
- Graph-based `veris affected` / `veris watch` — precise test selection via a new optional
  file-filter on the vitest/jest runners, with a conservative full-suite fallback.
- README/CHANGELOG updates; dogfood; `veriskit@0.3.0`.

### Out (deferred — do NOT build in v0.3)
- Framework route/endpoint mapping (Next.js/Express/etc.) → **v0.3.x**.
- Test generation → **v0.8**.
- Churn/git-history hotspots, HTML graph visualization, cloud sync.
- `explain`, `review`, `generate`, `repair`, risk *scoring* beyond the simple blast-radius signal.

## 4. Architecture

New modules only; existing modules consumed unchanged except the two runner adapters
(`vitest`, `jest`) which gain an **optional** file-filter, and `affected`/`watch` which
gain an optional graph path.

```
src/
  project-graph/
    model.ts            # ProjectGraph, ModuleNode types
    discover.ts         # walk the tree → candidate files, classify source/test/config
    scanner-resolver.ts # dep-free import extraction + relative resolution
    ts-resolver.ts      # project's typescript: preProcessFile + resolveModuleName
    graph.ts            # buildGraph(project) → picks resolver, assembles nodes + edges
    analyze.ts          # untested areas, blast radius, risky files
  cli/commands/
    scan.ts             # runScan(root) -> exit code
    plan.ts             # runPlan(root, {base?}) -> exit code
  runners/
    vitest.ts / jest.ts # + optional targetFiles on toCheck (run only those test files)
  affected/
    select.ts           # graph-based test-file selection (with conservative fallback)
```

## 5. Interfaces

```ts
// src/project-graph/model.ts
export type NodeKind = "source" | "test" | "config" | "other";
export interface ModuleNode {
  file: string;          // repo-relative, POSIX separators
  kind: NodeKind;
  imports: string[];     // resolved repo-relative intra-project files
  importedBy: string[];  // reverse edges
}
export interface ProjectGraph {
  root: string;
  resolver: "typescript" | "scanner"; // which resolver built this graph (honesty)
  nodes: Record<string, ModuleNode>;  // keyed by file
  sourceFiles: string[];
  testFiles: string[];
}

// src/project-graph/graph.ts
export function buildGraph(project: Project): Promise<ProjectGraph>;

// src/project-graph/analyze.ts
export interface Analysis {
  untested: string[];        // source files no test transitively imports (blast-radius desc)
  blastRadius: Record<string, number>; // file -> transitive dependents count
  risky: string[];           // high blast-radius ∧ (untested ∨ in `changed`)
}
export function analyze(graph: ProjectGraph, changed?: string[]): Analysis;

// src/affected/select.ts
export interface TestSelection {
  mode: "graph" | "full";    // "full" = conservative fallback (run everything)
  testFiles: string[];       // when mode==="graph": the specific test files to run
  reason: string;            // why full fallback was chosen (when mode==="full")
}
export function selectAffectedTests(
  graph: ProjectGraph,
  changed: string[],
): TestSelection;

// runners: toCheck gains an optional 3rd arg
// toCheck(project, cap, opts?: { targetFiles?: string[] }): Check
```

### Resolver selection (`buildGraph`)
1. If the project's `typescript` can be resolved from `project.root` **and** a `tsconfig.json`
   exists → use `ts-resolver` (`resolver: "typescript"`).
2. Otherwise → `scanner-resolver` (`resolver: "scanner"`).
Both produce the identical `ProjectGraph` shape. `scan`/`plan` print which was used.

### `ts-resolver` details
- Resolve `typescript` via `require.resolve("typescript", { paths: [root] })` (or dynamic
  import of that path). If it throws → fall back to scanner.
- Load compiler options from the nearest `tsconfig.json` via `ts.readConfigFile` +
  `ts.parseJsonConfigFileContent` (captures `paths`/`baseUrl`).
- Per discovered file: read text → `ts.preProcessFile(text, true, true)` → import specifiers.
  For each specifier: `ts.resolveModuleName(specifier, containingFile, options, host)`.
  Keep only `resolvedFileName`s **inside** `root` and not in `node_modules`.

### `scanner-resolver` details
- Read each `.ts/.tsx/.mts/.cts/.js/.jsx/.mjs/.cjs` file.
- Extract specifiers from `import … from "x"`, `import "x"`, `export … from "x"`,
  `require("x")`, and dynamic `import("x")` (string-literal argument only).
- Resolve only **relative** specifiers (`./`, `../`): try the literal path, then with each
  known extension, then `<path>/index.<ext>`. Ignore bare/aliased specifiers (documented
  limitation — the ts-resolver is the accurate path for alias-heavy repos).

## 6. Graph-based `affected` / `watch`

`selectAffectedTests(graph, changed)`:
1. If any changed file is **config/global** (`tsconfig*`, `package.json`, `biome.json`,
   `.eslintrc*`, `veris.config.*`, or a detected test-setup file) → `mode: "full"`,
   `reason: "config change affects the whole suite"`.
2. Else compute the set of **test files** transitively importing any changed file. If a
   changed file is **not present in the graph** (unresolved) → `mode: "full"`,
   `reason: "unresolved changed file"`.
3. Else → `mode: "graph"`, `testFiles: <that set>`.

`affected`/`watch` then:
- Compute capability gate as today (v0.2 `affectedChecks`).
- If `unit` is affected and `selectAffectedTests` returns `mode: "graph"` with a non-empty
  list → pass `targetFiles` to the unit runner (`vitest run <files>` / `jest <files>`).
- Otherwise run the unit check in full.
- `types`/`lint` remain project-wide.
- The scoped verdict (v0.1/v0.2) is unchanged; the terminal output notes when the unit run
  was narrowed (e.g. `unit  1.2s  (3 of 40 test files)`), so the user always sees that a
  subset ran — honesty about what executed.

**Honesty invariant:** narrowing happens ONLY in `mode: "graph"`. Any uncertainty →
full suite. A `verified`/`Affected checks passed` after a narrowed run means *every test
that could reach your change ran and passed*.

## 7. `veris scan`

Builds the graph, prints a calm map, writes `.veris/graph.json`:
```text
Veris — scan
Resolver    typescript          (or: scanner — no TypeScript found)
Modules     128   ·  Source 96  ·  Tests 32
Untested (top by impact)
  ! src/core/orchestrator.ts        23 dependents
  ! src/reporters/markdown.ts        9 dependents
High blast radius
  src/core/model.ts                 61 dependents
  ...
Graph        .veris/graph.json
```
Exit 0 (read-only analysis).

## 8. `veris plan`

Recommendations from the graph (analysis only, **no generation**). With `--base <ref>` it
also factors in changed files:
```text
Veris — plan
Test these first (high impact, untested)
  1. src/core/orchestrator.ts   (23 dependents, no test reaches it)
  2. ...
Verification setup
  ⚠ lint capability unavailable — no linter configured
Risky changes (vs HEAD)
  ! src/core/verdict.ts   changed · 41 dependents · untested
Next: `veris affected` runs only the tests reaching your changes.
```
Exit 0. Never writes source files; never claims a file is covered without a graph path to prove it.

## 9. Honesty (carried from v0.1/v0.2)

- Every `scan`/`plan` states the **resolver** used, so the user knows the graph's accuracy
  bound (the scanner can miss aliased/dynamic imports; TS resolver does not).
- "Untested" means *no test transitively imports this file in the built graph* — stated
  precisely, not as a coverage guarantee (a test importing a file ≠ the file being fully
  exercised; wording reflects that).
- Graph-`affected` narrows the unit run only under certainty (§6); the output always shows
  when a subset ran.
- `plan` recommends; it never generates or modifies code.

## 10. Milestones (subagent-driven, TDD)

1. **Graph model + discovery** — `model.ts`, `discover.ts` (walk + classify), tests on fixtures.
2. **scanner-resolver + graph assembly** — `scanner-resolver.ts`, `graph.ts` (scanner path), tests.
3. **ts-resolver** — `ts-resolver.ts` using the project's typescript; resolver selection + fallback, tests.
4. **analyze** — untested areas, blast radius, risky files, tests.
5. **`veris scan`** — command + report + `.veris/graph.json`, register, tests.
6. **`veris plan`** — command + recommendations (+ `--base`), register, tests.
7. **Graph-based `affected`/`watch`** — `select.ts` + runner `targetFiles` + wire into affected/watch with conservative fallback + narrowed-run output, tests.
8. **Docs + release** — README (scan/plan, graph-affected, resolver honesty), CHANGELOG `0.3.0`, version bump, dogfood.

## 11. Risks & mitigations

| Risk | Mitigation |
|---|---|
| Project's TS version/API differences | Use only stable, long-stable APIs (`preProcessFile`, `resolveModuleName`, `readConfigFile`, `parseJsonConfigFileContent`); wrap in try/catch → scanner fallback |
| tsconfig `paths`/`baseUrl` mis-resolution | Load real compiler options via `parseJsonConfigFileContent`; that's exactly what TS uses |
| Scanner misses aliased/dynamic imports | Documented limitation; ts-resolver is the accurate path; narrowing is gated to the typescript resolver, so scanner-mode graphs always run the full unit suite — misses can't hide tests because scanner mode never narrows |
| Narrowing a unit run hides an affected test | Conservative selection (§6): narrow only under certainty; config/unresolved → full suite |
| Large repos: graph build cost | preProcessFile (no type-check) is fast; walk respects ignores; cache the graph to `.veris/graph.json` (rebuild is cheap; no incremental cache in v0.3) |
| Monorepos / multiple tsconfigs | v0.3 targets a single project root (one tsconfig); document; monorepo-awareness deferred |

## 12. Explicit non-goals for v0.3

Framework route/endpoint mapping (v0.3.x) · test generation (v0.8) · churn hotspots · HTML
graph viz · incremental/persistent graph cache · monorepo multi-package graphs · `explain`/
`review`/`repair`. Anything here is refused in v0.3 with a clear message.
