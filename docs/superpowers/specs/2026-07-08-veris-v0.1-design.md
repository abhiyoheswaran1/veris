# Veris v0.1 — Design Spec

**Status:** Approved (design)
**Date:** 2026-07-08
**Scope:** v0.1 "Verification Foundation" only. Everything v0.2+ is explicitly out.

---

## 1. Thesis

Veris is a **verification control plane**, not a test runner. AI agents now produce
large diffs fast; the bottleneck has moved from writing code to *trusting* it. Existing
tools answer "did these tests pass?" Veris answers:

> Is this change safe enough to trust — and here is the evidence.

v0.1 wins on **unification, honest verdicts, and portable evidence** over tools people
already have. It does not out-execute Vitest; it wraps Vitest (and tsc, Biome, Jest,
node:test) behind one calm surface and turns a pile of green checkmarks into a
trustworthy verdict with a reviewable record.

### Strategic constraint (non-negotiable)

Veris does **not** build a native test execution engine in v0.1 — or at v1.0. It
orchestrates mature engines (Phase 1). It introduces clean internal interfaces so
engines can be swapped later (Phase 2). A native engine is Phase 3 only, and only where
it *provably* beats the tool it replaces after real usage. Premature reinvention is the
primary risk to avoid.

The moat is not the runner. It is the trust layer on top of it: the verdict, the
evidence, the orchestration, and (later) risk + failure explanation.

---

## 2. Locked decisions

| Decision | Choice | Rationale |
|---|---|---|
| Packaging | **Single npm package (`veris`)** with internal module boundaries named to match the future package split | Monorepo ceremony before a second consumer is overhead; internal folders sit behind interfaces and extract cleanly when real |
| Linter capability | **Auto-detect, no default** | Run ESLint or Biome only if the target repo has configured one; otherwise mark lint `skipped` with a reason. Never impose a linter |
| Verdict semantics | **Three-state: `verified` / `failed` / `partial`** | `partial` when capabilities were skipped. First-class honesty — never fold skipped into verified |
| Veris's own toolchain | **Vitest + Biome + tsup** | Modern, minimal deps; dogfoods Veris's own Vitest and Biome adapters as we build them |
| Dev harness weight | **Balanced — gate at milestone boundaries** | Real dogfooding + evidence trail without per-edit ceremony |

---

## 3. Scope

### In (v0.1 non-negotiables)

- Commands: `init`, `doctor`, `test`, `verify`, `report`
- Detection: project type, package manager, test scripts, runners (Vitest, Jest,
  node:test), TypeScript (`tsc`), linter (ESLint/Biome)
- Orchestrated execution with summarized terminal output; raw logs on disk
- Markdown verification reports
- Local `.veris/` store (runs, logs, reports, cache)
- Safe writes (never overwrite user files; idempotent `init`)
- Correct exit codes for CI
- Helpful, actionable errors
- README, CHANGELOG, LICENSE, release-ready package metadata
- Veris's own Vitest test suite

### Out (deferred — do NOT build in v0.1)

- Native test engine
- `watch`, `affected`, `scan`, `plan`, `explain`, `review`, `generate`, `repair`, `ui`
- Risk scoring / `risk`
- Browser execution (Playwright is **detected and reported as available**, not run)
- HTML / JSON reporters (a JSON shape may be stubbed internally, not shipped as a command)
- Plugin runtime (hook *names* documented; no loader)
- Cloud, auth, billing, telemetry, AI features

## 4. Architecture

Single package; folders named to match the future package split so they extract without
breaking public workflows. **The one interface that must be right on day one is
`Runner`** — everything else may churn.

```
veris/
  bin/veris              # thin entry -> cli
  src/
    cli/                 # arg parsing (cac), command routing, TTY detection
    core/                # run lifecycle, orchestrator, verdict engine, result model
    config/              # zero-config detection + optional config load
    project-graph/       # v0.1: shallow — pm/framework/script/runner detection only
    runners/             # Runner interface + vitest/jest/node-test/tsc/lint adapters
    reporters/           # terminal + markdown reporters (json shape internal only)
    evidence/            # .veris store: runs, logs, metadata
    util/                # exec (spawn), fs-safe, env, timing
  test/                  # veris's own tests + fixture repos
  docs/
```

### Dependency discipline

- ESM, Node >= 20 (dev on Node 26).
- Minimal deps: `cac` (arg parsing), `picocolors` (styling). Hand-rolled spinner + table.
  No chalk-weight or framework-weight dependencies.
- Lazy-load adapters only when their capability is active.
- **Cold-start budget: < 150ms to first paint.**

## 5. CLI UX

| Command | Behavior |
|---|---|
| `veris init` | Detect stack; write `.veris/config.json` only if absent; add `.veris/` gitignore entries; print what was found. Idempotent, never overwrites |
| `veris doctor` | Read-only environment + capability report: what is installed, what will run, and **what will be skipped and why**. No mutations |
| `veris test` | Run the detected test runner(s); summarized output |
| `veris verify` | Headline: run the configured check set (types, lint, unit) in parallel where independent; render the calm summary; write a report; exit non-zero on any failure |
| `veris report` | Render / open the latest (or `--last N`) report |

### Output contract

Matches the target aesthetic: project · changed · per-check status glyph + timing ·
verdict · report path. Raw logs live in `.veris/runs/<id>/` and surface only with
`--verbose` or on failure (last N lines inline; full log on disk). **Non-TTY / CI is
auto-detected** → plain, parse-stable output. Calm, readable, not noisy.

```text
Veris

Project     <name>
Changed     <n> files
Risk        —                (v0.1: not computed; shown as — )

Checks
✓ Types              4.2s
✓ Unit tests         8.7s
✓ Lint               2.1s
⊘ Browser smoke      skipped  (Playwright not configured)

Result
✓ Verified

Report
.veris/reports/verify-2026-07-08.md
```

`Risk` is displayed as `—` in v0.1 (risk scoring is v0.3+). Shown as a column so the
layout is stable across versions; never faked.

## 6. Data model (serializable — the spine of evidence)

```ts
Project        { root, packageManager, frameworks[], languages[], capabilities[] }
Capability     { id: "types"|"lint"|"unit"|"browser", available: boolean, reason?, runner? }
Check          { id, title, runner, cmd }
CheckResult    { checkId, status: "passed"|"failed"|"skipped"|"unknown", durationMs,
                 summary, logRef, counts? }
VerificationRun{ id, startedAt, project, results[], verdict, reportRef, env }
Verdict        { state: "verified"|"failed"|"partial", verifiedCapabilities[],
                 skipped[], reasons[] }
EvidenceItem   { runId, kind: "log"|"metadata"|"report", path, bytes }
Report         { runId, format: "md", path }
EnvironmentInfo{ os, node, pm, ci: boolean, timestamp }
```

`status: "unknown"` and verdict `partial` are first-class — this is how Veris never
manufactures confidence.

### Verdict rules

- Any `failed` check → `failed`.
- No failures, but ≥1 configured capability `skipped`/`unknown` → `partial`.
- All configured capabilities `passed` → `verified`.
- Exit codes: `0` verified, `1` failed, `2` partial (configurable to `0` via flag for
  teams that treat partial as acceptable; default `2` so CI is honest).

## 7. Runner strategy

`Runner` is the load-bearing interface. Adapters shell out via a central `exec` util
(array-arg `spawn`, no shell-string interpolation; resolves local bins from
`node_modules/.bin`; captures stdout/stderr with a timeout).

- Prefer machine-readable reporters where they exist: `vitest --reporter=json`,
  `tsc --pretty false`, `eslint -f json`. Biome via its JSON/summary output.
- Degrade gracefully to exit-code + captured-output parsing.
- **Parsing is enrichment over exit codes, never the source of truth for pass/fail.**

```ts
interface Runner {
  id: string;
  detect(project: Project): Promise<Capability>;
  run(check: Check, ctx: RunContext): Promise<CheckResult>;
}
```

Adapters in v0.1: `tsc`, `vitest`, `jest`, `node-test`, `lint` (eslint|biome). A future
native `VerisRunner` slots in behind this interface without breaking public workflows.

---

## 8. Evidence & reports

`.veris/` layout (runtime dirs gitignored; `config.json` may be tracked intentionally):

```
.veris/
  config.json
  runs/<id>/            # per-run raw logs + metadata.json
  reports/              # verify-YYYY-MM-DD.md
  cache/                # detection cache
```

Markdown report includes: project + env metadata, per-check status/timing/summary,
verdict with explicit skipped list + reasons, and log references. Suitable for pasting
into a PR. HTML/JSON reporters are v0.4.

---

## 9. Error handling

- Distinguish user errors (missing tool, bad config) from internal errors.
- Every error states what happened, why, and the next action.
- Never hide failures; never mark uncertain results as safe.
- Safe filesystem ops only: no destructive writes, idempotent `init`.
- Predictable CI behavior via correct exit codes.

## 10. Development harness (dogfooding workflow)

The three Baseframe tools are the harness we build Veris **inside** — they are **not
dependencies of Veris**. Veris remains fully standalone. Weight: **balanced — gate at
milestone boundaries**, not per-edit.

> Note: installed CLIs are authoritative; the marketing pages have drifted ahead of the
> shipped commands. Only commands verified present in the installed CLIs are used.

| Concern | Tool of record | Commands (installed, verified) | When |
|---|---|---|---|
| Task discipline + done-ness gate | **AgentLoopKit** 0.36.2 | `create-task`, `verify`, `check-gates`, `ship`, `handoff` | Open + close of each milestone |
| Session flight recording | **AgentFlight** 0.5.1 | `start`, `snapshot`, `report` | Session start; snapshot per milestone boundary |
| Structural risk / review lens | **ProjScan** 4.9.3 | `doctor`, `review`, `hotspots`, `coupling`, `impact` | `doctor` at init; `review` before each commit |

**Ground-truth verification** is plain npm scripts (Vitest + Biome + `tsc --noEmit`).
AgentLoopKit `verify` captures that run as gate evidence; AgentFlight `snapshot` records
it into the timeline. No double-verify.

### Per-milestone loop

```
agentloop create-task        # scope: outcome, acceptance, protected files
agentflight snapshot         # timeline start
  -> TDD: red -> green -> refactor      (superpowers:test-driven-development)
  -> npm run verify                     # vitest + biome + tsc --noEmit (ground truth)
projscan review              # structural risk of the diff, pre-commit
agentloop verify             # capture verification as gate evidence
agentloop check-gates        # task + verification + git gates pass?
  -> commit
agentflight snapshot         # milestone complete in flight log
```

At v0.1 close: `agentloop ship` + `handoff`, `agentflight report`, final `projscan review`.

### Superpowers skills

`brainstorming` (this spec) → `writing-plans` (next) → per-milestone
`test-driven-development` + `verification-before-completion` → `requesting-code-review`
→ `finishing-a-development-branch`.

## 11. Milestones

Each is independently shippable and dogfoodable.

1. **Scaffold** — package metadata, TS strict, Biome, Vitest, tsup, `bin/veris` prints
   version. Harness init (`agentloop init`, `agentflight init`, `projscan doctor`).
2. **Detection + `doctor`** — pm/framework/script/runner detection, capability model,
   read-only report + fixture repos.
3. **Runner interface + first adapters** — `tsc` and Vitest; `exec` util; `.veris/runs`.
4. **`test` + terminal reporter** — the calm summary surface.
5. **`verify` orchestrator** — parallel independent checks, verdict engine, exit codes.
6. **Markdown reporter + `report`** — evidence write + render.
7. **`init` + config** — safe writes, gitignore management, optional config load.
8. **Remaining adapters (Jest, node:test, ESLint/Biome), error-handling pass, docs,
   CHANGELOG, dogfood on a real repo → cut v0.1.**

---

## 12. Risks & mitigations

| Risk | Mitigation |
|---|---|
| Heterogeneous tool output brittle to parse | Prefer JSON reporters; parsing enriches, exit codes decide |
| "Verified" overclaims | Verdict engine treats absent capabilities as `skipped`/`partial`, surfaced loudly |
| Cross-platform shelling (quoting/PATH/Windows) | Central `exec`, array-arg spawn, resolve `node_modules/.bin` |
| Scope creep into v0.2+ | Section 3 "Out" is a contract; anything not in "In" is refused |
| Cold-start bloat | Dep budget + lazy-load adapters per active capability |
| Premature monorepo/native-engine tax | Single package behind interfaces; native only Phase 3 where proven |

- [ ] `package.json`: name, version `0.1.0`, `bin`, `exports`, `files`, `engines`, repo/license

## 14. Explicit non-goals for v0.1

Native execution · risk scoring · `watch`/`affected`/`scan`/`plan`/`explain`/`review`/
`generate` · browser execution (detect-only) · HTML/JSON reporter commands · plugin
runtime · cloud/auth/billing · AI features. Anything here is refused in v0.1 with a
clear message rather than half-built.
