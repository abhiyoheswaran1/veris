# VerisKit v0.4.1 — Design Spec ("Signed Evidence")

**Status:** Approved (design)
**Date:** 2026-07-10
**Branch:** `feat/veris-v0.4.1`
**Builds on:** v0.4 (shipped, `veriskit@0.4.0`). Reuses the canonical record + digest (`evidence/record.ts`), the verify path (`evidence/verify-evidence.ts`), the bundle (`evidence/bundle.ts`), and the `evidence <action>` CLI dispatcher.

---

## 1. Thesis

v0.4 gave every run a git-anchored `evidence.json` with a sha256 **integrity digest**, labeled honestly as "detects edits and corruption, not forgery-proof on its own." v0.4.1 closes that gap: anyone can **cryptographically sign** an evidence record and verify the signature, so the record proves not just that it is intact but that a specific key vouched for it.

Signing uses node's built-in **Ed25519** (`node:crypto`), so it adds no runtime dependency, works offline, and runs in CI. Still exactly two runtime dependencies (`cac` + `picocolors`). Still honest: a signature proves the holder of a private key signed the record, not *who* that holder is; VerisKit says so and prints the key fingerprint.

Signing is **opt-in and additive**. Unsigned evidence still verifies for integrity exactly as in v0.4.

## 2. Locked decisions

| Decision | Choice | Rationale |
|---|---|---|
| Mechanism | **Ed25519 via `node:crypto`** | Real asymmetric signing, zero new deps, offline, CI-friendly, always available |
| Sigstore keyless | **Deferred** | Needs the sigstore dependency + network + OIDC; breaks the two-dep budget |
| What is signed | **The record `digest` string** | The digest already commits to the whole canonical record; signing it is equivalent and simpler |
| Signature storage | **Detached sidecar `<evidence.json>.sig`** | Never alters the record's own digest (no chicken-and-egg); portable |
| Public key | **Embedded in the signature** | Verify is self-contained; trust is still established out of band (SSH/minisign model) |
| Private key at rest | **Unencrypted PEM, `chmod 0600`, no passphrase prompt** | CI-usable; VerisKit never handles a passphrase. Passphrase support is a later addition |
| keygen default location | **`.veris/keys/`, gitignored by `init`** | A private key cannot be committed by accident. `--out` overrides |

## 3. Scope

### In
- `src/evidence/signing.ts`: `generateKeyPair()`, `signDigest()`, `verifySignature()`, `keyId()`, the `Signature` type. All `node:crypto`.
- `veris evidence keygen [--out <path>]` — Ed25519 keypair, private `chmod 0600`, prints fingerprint + a keep-it-secret warning, refuses to overwrite an existing key.
- `veris evidence sign <evidence.json> --key <priv> [--out <sig>]` — recompute+check the record digest, sign it, write `<evidence.json>.sig`. Private key may also come from `VERISKIT_SIGNING_KEY` (PEM contents) for CI.
- `veris evidence verify <evidence.json>` extended: checks a sibling `.sig` (or `--sig <path>`) when present; `--pubkey <path>` / `--key-id <id>` asserts the expected signer. Exit 1 on any digest or signature failure.
- `veris evidence bundle` extended: fold an existing `.sig` into the bundle so it stays self-verifying.
- `veris init` gitignores `keys/`.
- README + CHANGELOG; dogfood; publish `veriskit@0.4.1`.

### Out (deferred)
- Sigstore keyless / OIDC / transparency log.
- A named trust store (fingerprint to human name mapping); v0.4.1 uses out-of-band fingerprint comparison and `--pubkey`/`--key-id` assertion.
- Passphrase-encrypted keys, key rotation, multiple signatures per record.
- Signing during `verify`/`affected` automatically; signing stays an explicit step.

## 4. Architecture

New module plus extensions to three existing evidence modules and the CLI dispatcher.

```
src/
  evidence/
    signing.ts          # Signature type; generateKeyPair; signDigest; verifySignature; keyId
    verify-evidence.ts  # (extended) load + check a sibling .sig; assert expected signer
    bundle.ts           # (extended) optional signature carried + verified
  cli/commands/
    evidence.ts         # (extended) keygen + sign actions; verify shows signature result
  cli/index.ts          # (extended) evidence <action> accepts keygen | sign; new options
```

### 4.1 The signature object (`veriskit/signature@1`)

```jsonc
{
  "schema": "veriskit/signature@1",
  "alg": "ed25519",
  "keyId": "a1b2c3d4",              // first 8 hex chars of sha256(public key DER)
  "publicKey": "<base64 SPKI DER>", // the Ed25519 public key
  "digest": "sha256:...",           // the record digest that was signed
  "signature": "<base64>",          // ed25519 signature over the UTF-8 bytes of `digest`
  "signedAt": "<iso timestamp>"
}
```

- `keyId = sha256(publicKeyDer).slice(0, 8)` (hex). Short fingerprint for display and `--key-id` assertion.
- The signed message is the UTF-8 bytes of the record's `digest` string (for example `sha256:9f2c...`).
- Ed25519 needs no hash algorithm argument: `crypto.sign(null, msg, privateKey)` and `crypto.verify(null, msg, publicKey, sig)`.

### 4.2 signing.ts interfaces

```
generateKeyPair(): { publicKeyPem: string; privateKeyPem: string; keyId: string }
keyId(publicKey: KeyObject | string): string          // sha256(DER).slice(0,8)
signDigest(digest: string, privateKeyPem: string): Signature
verifySignature(sig: Signature): boolean               // sig valid for its embedded key
```

`verifySignature` checks the signature against the embedded public key only. Asserting *which* key (via `--pubkey`/`--key-id`) is done in the verify command layer by comparing `keyId` / the embedded key bytes.

### 4.3 verify integration

`verifyEvidenceFile(path, opts?)` gains optional `opts` for `{ sigPath?, expectedPubKey?, expectedKeyId? }`. When a sibling `<path>.sig` exists (or `sigPath` given) it adds signature checks to the `VerifyResult.checks`:

- `signature`: the signature is valid for its embedded public key AND its `digest` equals the record digest.
- `signer` (only when `--pubkey`/`--key-id` supplied): the embedded key matches the asserted key.

`VerifyResult` gains `signed: boolean`. Overall `ok` stays "all checks pass". A missing signature adds no check and is not a failure; a present-but-invalid signature fails.

### 4.4 bundle integration

`Bundle` gains optional `signature?: Signature`. `buildBundle(record, report, logs, signature?)` includes it and adds `manifest.signature` (sha256 of the canonical signature) so `verifyBundle` checks it under `bundleDigest`. `verifyBundle` runs the signature checks when a signature is present.

## 5. Honesty guarantees (the brand)

- A signature proves the holder of a private key signed the record. It does **not** prove identity. `verify` prints the key fingerprint and, without `--pubkey`/`--key-id`, states that the verifier must confirm they trust that key.
- Unsigned evidence verifies for integrity exactly as before; signing never becomes silently mandatory.
- VerisKit never transmits the private key and never writes it anywhere except the path the user chose; keygen sets `0600` and warns.
- The private key is never logged, never embedded in any record, bundle, or signature (only the public key is embedded).

## 6. Testing

- **keygen:** produces a valid Ed25519 pair; the private file is `0600`; refuses to overwrite an existing key; printed `keyId` matches `sha256(pub).slice(0,8)`.
- **round-trip:** `signDigest` then `verifySignature` passes; a tampered `signature` fails; a tampered `digest` in the sig fails.
- **verify integration:** signed record with a valid sig passes; tampering the record body fails the digest check (not the signature); tampering the sig fails the signature check; `--pubkey`/`--key-id` mismatch fails the signer check; an unsigned record still verifies for integrity (no signature check, `signed: false`).
- **bundle:** a bundle carrying a signature verifies; tampering the embedded signature fails.
- **CLI:** `keygen` writes both files and a fingerprint; `sign` writes `.sig`; `verify` exit codes (0 all-pass, 1 any-fail); `VERISKIT_SIGNING_KEY` env path works.

## 7. Deliverable

`veriskit@0.4.1`: `veris evidence keygen | sign`, signature-aware `verify` and `bundle`, honest trust labeling, README + CHANGELOG updated, dogfooded (generate a throwaway key, sign a real run, verify it), published via the existing OIDC release pipeline.
